# SOFT SKILLS ソフトウェア開発者の人生マニュアル 第1部
[全体目次へ戻る](index.md)

## 第1部 キャリアを築こう
ソフトウェア開発のキャリアで頭角を現すためには何を望むべきか、どのようにしてそれを手に入れるかを判断するプロセス

### 2. スタートから派手にいこう!: 誰もがするようなことはするな
- 会社との関係を捉え直す: ソフトウェア開発のキャリアを｢事業(ビジネス)｣として扱う。マインドセットを、雇用契約で縛られた奴隷から、自分自身の事業を経営している事業者に切り替える

- 自分の提供できるサービスとは? → ソフトウェア開発(アイデアを受け取ってそれをデジタル化された現実に変身させる能力)
  + この｢サービス｣を絶えず改良していかなければならない
  + 自分が提供できる｢サービス｣は、そこらのソフトウェア開発者のものと価値がどのように違うのか?を伝えなければならない(マーケティング)
    * マーケティングがうまければうまいほど、その料金を高くでき、多くの顧客を惹きつけられるように

- 大半の駆け出しソフトウェア開発者と同じにならないためには?
  + 自分が提供するサービスに全力を集中、その売り方を考える
  + サービスを向上させる方法を考える
  + 特定の顧客やニーズに応えられるようにサービスを専門化する方法を考える
  + 特別の顧客を対象に専門的なサービスを提供するスペシャリストになることに力を注ぐ

- 自分のサービスを広く知ってもらう方法を学ぶことも大切

### 3. 未来について考える: あなたの目標は何?
- 次のステップ: ビジネスの目標をはっきりさせる
  + 目的地を定めずに踏み出した一歩一歩は、すべて無駄足である
- 目標の設定方法: まず大きな目標を描き、それに到達するための小さな目標を途中に設けていく
  + 大きな目標…自分のキャリアで最終的に何がしたいのか? 何を手に入れたいのか? 5~10年後の自分をどうしたいのか?
  + 小さな目標…キャリアの最終形に達するまでの道筋 大きな目標から現況を逆に辿ってみる
  + 例) 大目標: 体重を45kg落とす → 小目標: 2週間ごとに2.3kg減量する
- 小さな目標を設けていないと、軌道から外れたときに修正できない。小さな目標があれば、モチベーションの維持にも役立つ

- 目標は定期的に見直して更新するようにする。週･月･季節･年ごとに、設定した目標をレビューし、次の目標を設定する。

### 4. 社交術: 考えている以上のものが必要だ
- ソフトウェア開発者の仕事は、ただコードを書くことではない。他の職種と同様、人と接することである
  + コードでさえ、第一義的には人に理解してもらうために書くものである

- どんな人でも、｢大切にされている｣と感じたい
  + 人を見くびったり、存在や達成を貶めたりすれば、相手は凶暴な反応を返してくる → 同僚のアイデアをはねつけたりしてプライドを傷つけてはいけない

- 批判は結果を生み出す助けにはならない
  + いい行動を褒めることの方が、悪い行動を懲らしめるよりはるかに効果的
  + 部下にベストの力を出してもらうには、余計なことをいわずに励ましの言葉だけを口にせよ

- 相手にとって大切なこと、その人が望んでいることを考える
  + 同僚･上司と初めて対話するときには、話題の中心を相手に持っていくようにする。｢この対話から相手が得ようと思っていることは?｣と考える。
  + 対話がどのように進むか、事前に準備しておくことも大切

- 議論を避ける
  + 人間は感情的な生き物である。冷たくされた同僚は、自分のやり方を認めてはくれない
  + 何かのやり方について賛成できないとき、それが命をかけるに値するほどの問題かを判断する。他人にとっては大きな意味を持つ小さな問題で自分の意見を諦める機会を設ければ、相手を計り知れないほど尊重した、という実績を作れる → 立場が逆転したとき、相手が考えてくれるかも…

- どんなにがんばってもうまくやっていけない人に対しては、接触を避けるに越したことはない

### 5. 面接をハッキングするコツ
- 面接で｢合格｣するための手っ取り早い方法…面接官に気に入ってもらう
  + ほとんどの面接官は、技術以外のあらゆる要素に基づいて人を採用する。スキルは低くても、好感の持てる人が最後は勝つ
- 面接が始まる前に、面接のための戦略について考えておく。応募する会社内に接点を築くことが必要
  + 個人推薦をもらっておく。推薦を受けて面接に向かうと、推薦者の社会的信用を借りることになるので、面接官は自分を自動的に高く評価してくれる
  + 応募する会社に知人がいない場合は? その会社で働いている開発者のブログを見つけ、関係を築く
- 本物の面接では…｢求められていることへの理解｣｢それを実行できることに対する自信｣を示す
  + 技術的に優秀であることを示した上で、自律的に仕事を進められる開発者であると面接官に思わせる

- まとめ
  + まず技術的なスキルを磨いておく
  + 人脈づくりに着手…企業の社員との間に接点を作っておく
  + 面接の練習を繰り返す

### 6. 雇用形態: 三つの選択肢を理解する
- ソフトウェア開発者としてのキャリアには、就職以外にも選択肢があることを知っておく

1. 会社の従業員…多くの開発者が選ぶ自明な選択肢
  + 利点: 安定していること。少なくとも一定期間にわたって決まった水準の収入に依存できる。また、道筋がはっきりしているため、選びやすい。有給休暇がある。健保･年金で有利
  + 欠点: 自由がない。会社員でいる限り、収入が頭打ちになるときがくる

2. 独立系コンサルタント…特定の雇用主のためではなく、1件以上のクライアントのために仕事をし、収入の多くを稼ぐ開発者
  + 契約プログラマーとは大きく異なる
  + 利点: 時間の使い方･仕事の選択の自由を得られる。高い収入を得られる可能性がある
  + 欠点: 自分で仕事を探さなければならない。事業経営に関連するあらゆることを自分でしなければならない。結局上司がたくさんになるだけかも…

3. アントレプレナー…最も難しく、最も不定形で、最も高い収入が得られる可能性がある
  + ソフトウェア開発のアントレプレナーとは、｢ソフトウェアのスキルを活用して独自の製品を開発する人｣
  + 会社員は時間とお金を交換 アントレプレナーは時間と将来の莫大な払い戻しのチャンスを交換…最初のうちは無償で働いている
  + 利点: 完全な自由･完全に天井無しの収入の可能性。自分以外の上司はいない
  + 欠点: 完全に自己責任。必要なスキルを手に入れるためにかなりの投資をする必要がある

- どれを選ぶべきか? 初めて仕事をする場合は、会社の従業員になることは十分意味がある

- 独立系コンサルタントやアントレプレナーになりたければ、まずそれをしている知人に会って話を聞くこと

### 7. あなたはどのタイプのソフトウェア開発者か
- 専門性は重要。専門を持つことによって、チャンスの数が減るのは確かだが、かわりにチャンスを獲得する可能性は高くなる
- どこまで専門的になればいいのか→場合による
  + 例: プリンター言語を専門とする開発者…かなり専門領域を絞った言い方→ごく少数の就職先候補にとって極めて価値の高い人間
    * 市場を世界全体に広げれば、かなり狭い専門であっても需要は大きい。
    * 逆に、働きたい場所が限られる場合は、専門を狭く持ちすぎると仕事がない…

- 最初にはっきりさせるべきなのは、どんなタイプのソフトウェアを開発したいか
  + UIの作成といったフロントエンド
  + ビジネスロジックを実装するミドルウェア
  + データベースや低水準処理を相手にするバックエンド
    * これらを全てカバーするかわりに、特定の開発スタックに特化することもできる
  + 組込みシステム
  + オペレーションシステム
  + モバイルアプリケーション開発→特定のモバイルOS(iOS, Android)
  + 限定されたプラットフォームやフレームワークの専門家

- 専門分野を選ぶためのヒント
  + 過去の会社で感じた｢ペインポイント｣を解決することを専門とする人間になれるか?
  + 誰もやりたがらない仕事・スキルのある人がいない仕事はあるか?→その分野を専門にしてしまえ
  + カンファレンスやユーザーグループでよく話題になるテーマ
  + 同僚やStack Overflowなどのサイトで自分がよく答える質問は?

- 専門はあとでいつでも変えられる
- 複数のテクノロジー･多くのプログラミング言語を扱えるようにしておくと、限られたテクノロジー･言語しか知らない人よりも貴重な、柔軟性のある開発者になれる
  + が、何でも屋として自分を売り込むのは難しい
  + まず専門特化に取りかかり、あとで多彩になればいい

### 8. ｢どの会社も同じ｣ではない
- 長い目で見ると、職場環境は自分にとって重要な意味を持つことになる。大･中･小の会社それぞれの長所･短所を検討する

- 小さな会社: 大半はスタートアップ
  + 急成長することに全力をかけ、差し迫った目標を達成するためなら何でもする
  + このような会社の中では、ただコードを書いていればいいというわけにはいかない。より柔軟になる必要がある
  + 自分のすることが会社に大きな影響を及ぼす。偉大な達成をすれば/ヘマをすれば、それがより目立ちやすい
  + 大企業よりも安定性では劣るが、報酬は大きい。出世の可能性は高くなる
  + ペースが早くテンションが上がる環境が好きで、何かを作りあげるための一員となり、それが成長するのを見てみたいなら、小企業を選ぶと良い

- 中規模の企業
  + 小企業よりも職務の定義がはっきりしており、安定度も高い。大企業よりも安定しているかも
  + じっくり堅実なタイプの人が競争に勝つ場合が多い
  + リスクを避けがちなので、最先端の仕事をしたい場合は上司に評価されるのが難しくなる

- 大企業
  + 個々の大企業は、互いに大きく異なる
  + 仕事上、決められた手順や手続きが多い。手順や階層構造が好きなら楽しく働ける?
  + 与えられるチャンスが大きい。様々な訓練を受けられ、ほとんどすべてのソフトウェア製品を使わせてもらえる。優れたスタッフと仕事をするチャンスも得られる
  + 大規模で世界を変えるようなプロジェクトに参画できる。逆に自分の貢献はプロジェクトに大きな影響を与えない
    * ソフトウェアシステムのあらゆる側面に関わりたいタイプの開発者にとってはつまらない
  + 大きな政府に匹敵するほどの複雑な政治システムが存在することが多い。出世のためには、会社の政治的環境を泳ぎ渡る方法を学ばなければならない

- ソフトウェア開発会社(主要な事業としてソフトウェア開発を行う会社)と、ソフトウェア開発者がいる会社(社内ソフトウェアの開発のために開発者を雇っている会社)があつ
  + 後者のような会社では、ソフトウェア開発者は一般にあまり大切にされない
  + ソフトウェア開発会社では、開発者に高い価値をおいている場合が多い。最先端の技術を取り入れている会社も多い

- 現実には、会社ごとに独自の文化がある。採用通知を承諾する前に、会社の中の人にこのような点について聞いておくとよい

### 9. 出世階段の上り方
- 世の中には、浮かび上がることが永遠にないのではないかと思えるような人がたくさんいる。そんな人間になりたくなければ、そのための行動を起こさなければならない。
- 地位を上げるためにできる最も大切なことは、重い責任を引き受けること。収入は責任に追いついてくる
  + 自分で責任を引き寄せるためにできることは? あちこちを探ってみて、自分の能力を活かせる場所を見つければいい
  + 誰もやりたがらない仕事をあえて引き受けて成果を出すことで、周囲に自分の真価を示すことができる
  + 他のメンバーのメンターになるという方法もある。他人の問題を解決すると、自分がより多く学べるだけでなく、チーム内で頼れる人だという評判が生まれ、チームリーダーになれるチャンスが…

- 自分がどんなに優れた仕事をしても、それが周囲に知られなければ意味がない。自分の存在を主張するためにできることはたくさんある
  + 毎日の仕事の記録を残し、それをまとめた週報を上司に送る
  + 発表や訓練の講師をすると申し出る
  + 声を上げる
  + 上司との定期的なミーティングを設定し、積極的に｢見られ｣に行く

- スキルと知識を磨き続ける。勉強を続ければ、以前よりも価値のある人間になっていることがはっきりとわかるので、昇給の理由が簡単に作れる
  + 資格認定を目指すのもよい。資格試験のために努力していることにより、上司は自分のことを認めてくれる
  + ソフトウェア開発以外のことも勉強しよう。経営陣に入る可能性が出てきたら、リーダーシップ･経営･マネジメントについても学ぶべき
  + 学習している事実を知ってもらうことを忘れてはならない。発表のほかにも、ブログを開設したり、書籍を執筆したり、カンファレンスで講演したりと自分が勉強したことをアピールする方法はたくさんある

- 問題に対する答えをいつでも出せる人、その答えを実行して結果を出せる人になろう

- 社内政治に対する対処はもっとも重要性が低い。巧妙な先述を駆使して出世階段を駆け上ることはできるが、同じくらい簡単に転げ落ちる危険がある。本当に価値のある人間になったほうがいい結果になる

- 昇進昇格のチャンスが全く与えられない場合は、辞めてしまえばいい。その前に新しい就職先を確保しておくこと

### 10. プロであること
- プロであるということは、自分の姿を見せ続け、仕事を進め、前進し続けること。プロになるには、自分の問題点を克服し、可能な限り最高の仕事を生み出さなければならない
- プロとアマチュアの違い

|プロ|アマチュア|
|:---|:---------|
|常に従う原則を持っている|頼まれたことは何でもする|
|仕事を **正しく** 終わらせる|仕事をとにかく終わらせる|
|間違っていること、知らないことを認めるのを恐れない|知ったかぶりをする|
|首尾一貫している|予測不能で信頼できない|
|責任を取る|責任を避ける|

- 自分がプロになるには? プロの習慣を身に付ける必要がある。
  + たとえば、朝礼で自分が言う予定の内容を事前に準備するという習慣
  + 時間管理能力: 今日は何の仕事をするのか、ルーチン作業にどれくらい時間がかかるかを把握できているか? 1日のプランをしっかり練ること
  + 習慣は安定性を築き、安定性は自分に対する信頼を生むので、重要 cf) 『習慣の力』byチャールズ･デュヒッグ

- 技術的･倫理的に難しい問題にぶつかったとき、正しい選択ができなければならない
  + 技術的問題には、正しい解決方法がある
  + 倫理的問題には、いつも明快な正解があるとは限らない。クライアントにとって大きな利益が得られるが、自分たちの幸福が危険に晒されるようなとき、その決定を推し進められるか?
    * ｢ノー｣と言うことの重要性: 自分が正しいと思っていることを一貫して選んだほうが、報われる可能性が高い

- プロとして、自分が作るコードの品質を向上させるために常に努力しなければならない
  + 多くの開発者は、品質が基準に達しないときに、基準をクリアする方向ではなく、基準を引き下げる方向に向かってしまいがち
  + 品質基準は、コードの隅々まで徹底することが重要。どこかで妥協をすると、元に戻るのは難しい
  + プロになるには、いつも全力で自分のスキルを向上させ、多くのことを学ばなければならない。常にもっといい自分を求め続けよう

### 11. 自由を得る: 仕事の辞め方
- 会社を辞めて自力で働きたいなら、辞める前に現実的なプランを練っておかなければならない
  + 辞めてから立ち上げたいと思っている事業を副業として始め、生活を支えられるくらいの収入が得られる見込みが立つまでは、会社を辞めずに仕事の比重をずらしていくようにする

- 独立して働くための準備として、新しい仕事を副業として始め、軌道に乗せておくことが大切
  + 自分のために働くのは非常に大変。どれだけの仕事が必要かを事前に知っておく必要がある
  + 副業の成否次第で生き残れるか決まるような事態にはならないので、リスクを避けられる
  + 新事業は失敗する可能性が高い

- 自分のために働くときには、他人のために働くときよりも一所懸命働くことになるので、重い負荷に慣れておく必要がある
  + 他人のために仕事をしているときと、自分のために仕事をしているときの仕事の質の違い: 街をジョギングしているときと、ライオンに追いかけられて必死に走っているときの違い
  + 自分のための仕事に対する情熱は時間とともに失せていく
  + 自分のために働くときには、8時間分の仕事ができるとは限らないと思っておく

- 副業を始める前に、会社の就業規則を必ずチェックすること
  + 副業に使った時間とリソースについて記録を残し、成果物の帰属に関する問題が起きないようにしておく
